# W6.5 — End-to-end TTS demo (caller dials → hears bot audio)

## Goal

Prove W6 end-to-end by making a SIP phone (linphone on the user's Mac)
dial into a Docker-hosted FreeSWITCH instance and hear bot speech
synthesised by the existing open-tts engine, via the new
`mod_open_switch.so` `StartTts` RPC.

This is **not a wave** — it is a sample / integration smoke that
exercises the full stack. It lands after W6 Track C merges and
before W7 sub-agents kick off (so W7 doesn't have to rebase against
demo changes).

## Components to build

### 1. tts-edge `MediaBridge` adapter (open-tts repo)

**Path**: `tts-edge/internal/bridge/openswitch/`

A new bridge alongside the existing
`tts-edge/internal/bridge/freeswitch/` legacy `socket()` adapter.
The legacy adapter stays for backwards-compat; the new adapter speaks
the gRPC `open_switch.media.v1.MediaBridge` schema.

**Files** (~400 LOC):

```text
tts-edge/internal/bridge/openswitch/server.go         (gRPC service registration)
tts-edge/internal/bridge/openswitch/stream_handler.go (one bidi stream per call)
tts-edge/internal/bridge/openswitch/synthesis_proxy.go (StartSession → ... mapping)
tts-edge/internal/bridge/openswitch/frame_codec.go    (PCM-S16LE re-frame to 20ms @ 8/16kHz)
tests/integration/openswitch_bridge_test.go           (smoke test)
```

**Stream handler responsibilities**:

1. Receive `FromModule.start` (must be the first message; else error
   `INVALID_ARGUMENT`).
2. Branch on `start.purpose`:
   - `TTS_PLAYBACK`: open an internal `RealtimeTts.Synthesize` stream;
     send `StartSession{tenant_id, voice_id}` then
     `StartTurn{} + AppendText{start.start_message} + CommitText{}`.
     The voice_id comes from `start.variables["voice_id"]` (operator
     decides) or a compose-level default.
   - `STT_TRANSCRIBE`: stub — return `Control{error: UNIMPLEMENTED}`.
   - `VOICEBOT_DUPLEX`: stub for now (V2).
   - `RECORDING_RELAY`: out of scope for the demo.
3. Drain `TtsEvent.audio_frame` from the engine; re-frame to
   `start.sample_rate_hz` (typically 8000 for FS G.711); emit
   `FromService.audio = AudioFrame{seq, ts_samples, duration_samples,
   payload_pcm_s16le}` at 50 fps (20 ms each).
4. Half-close on `FromModule.control{op: HALF_CLOSE_SEND}` or stream
   client cancel.
5. Map errors to gRPC status codes.

The internal Synthesize call uses the existing engine path (no engine
changes needed — `voice_id` resolution + fixture-mode token-gen
already work in V0).

### 2. FreeSWITCH Docker image with `mod_open_switch.so` installed

**Path** (already exists in skeleton form): `deploy/docker/`

The repo already has `deploy/docker/Dockerfile.builder` that builds
the module. The demo needs a **runtime** image that:

1. `FROM simplefs/open-switch-base:1.10.12-trixie` (already pushed).
2. `COPY mod_open_switch.so /usr/local/freeswitch/mod/`
3. `COPY conf/freeswitch/* /etc/freeswitch/` — bare-minimum FS config:
   - `freeswitch.xml` — load `mod_sofia`, `mod_dptools`, `mod_lua`,
     `mod_open_switch`.
   - `autoload_configs/modules.conf.xml` — explicit module list.
   - `dialplan/default.xml` — one extension `9100` (see §3).
   - `sip_profiles/internal.xml` — port 5060, listen 0.0.0.0,
     no auth (demo only).
4. `EXPOSE 5060/udp 5061/udp 16384-32768/udp 50071/tcp`

**Files** (~150 LOC):

```text
deploy/docker/Dockerfile.runtime       (the runtime image)
deploy/demo/freeswitch.xml             (FS root)
deploy/demo/modules.conf.xml           (module list)
deploy/demo/dialplan/default.xml       (ext 9100)
deploy/demo/sip_profiles/internal.xml  (SIP profile)
deploy/demo/open_switch.conf.xml       (module config — TTS endpoint)
deploy/demo/lua/start_tts.lua          (dialplan glue — calls StartTts)
deploy/docker-compose.demo.yml         (full stack)
```

### 3. Dialplan — extension 9100

```xml
<extension name="osw-demo-tts">
  <condition field="destination_number" expression="^9100$">
    <action application="answer"/>
    <action application="sleep" data="500"/>
    <action application="set" data="osw_demo_message=Xin chào, đây là demo của mod_open_switch."/>
    <action application="lua" data="start_tts.lua"/>
    <action application="park"/>
  </condition>
</extension>
```

`start_tts.lua` uses `mod_lua` + a tiny Lua gRPC client (or shells out
to `grpcurl`) to invoke the module's own `StartTts` RPC. Body:

```lua
-- start_tts.lua
local channel = session:getVariable("uuid")
local message = session:getVariable("osw_demo_message") or "Hello demo."

-- Use grpcurl as a quick path (no Lua gRPC dep). The module's control
-- listener is on 127.0.0.1:50071 inside the container.
local cmd = string.format(
  "grpcurl -plaintext -d '{\"channel_uuid\":\"%s\",\"upstream_endpoint\":\"tts-edge:50061\",\"sample_rate_hz\":8000,\"start_message\":%q,\"header\":{\"tenant_id\":\"demo\",\"request_id\":\"%s\"}}' 127.0.0.1:50071 open_switch.control.v1.Control/StartTts",
  channel, message, channel
)
local rc = os.execute(cmd)
freeswitch.consoleLog("INFO", "start_tts.lua: StartTts rc=" .. tostring(rc) .. "\n")
```

(Production setups would write a tiny Go/C++ client or use FS's
`mod_curl_grpc`; for the demo, `grpcurl` is sufficient.)

### 5. Linphone config on the Mac host

```text
SIP identity:    sip:1000@127.0.0.1:5060
Transport:       UDP
Outbound proxy:  127.0.0.1:5060
(No auth — internal profile)
Dial:            9100
```

When the user dials 9100:

1. Linphone INVITE → docker FS port 5060 (mapped on host).
2. FS answers, runs dialplan, executes `start_tts.lua`.
3. Lua shells out to `grpcurl` with the channel UUID.
4. Module's `StartTts` handler runs: attaches WRITE_REPLACE bug,
   opens bidi gRPC stream to `tts-edge:50061`.
5. tts-edge maps to internal Synthesize, engine emits audio bytes.
6. Module receives `FromService.audio`, pushes through TtsPlayoutBuffer,
   bug callback injects into FS write side.
7. Linphone plays the audio out the Mac speakers.

User hears: "Xin chào, đây là demo của mod_open_switch."

## Implementation order

1. **Step 1** — open-tts: build `tts-edge/internal/bridge/openswitch/`
   adapter (Sonnet sub-agent in open-tts worktree). ~2 hours.
2. **Step 2** — open-switch: build `deploy/docker/Dockerfile.runtime` +
   demo `conf/freeswitch/*` (Sonnet sub-agent in open-switch worktree).
   ~1 hour.
3. **Step 3** — `docker-compose.demo.yml` + lua glue script
   (orchestrator-side, lightweight). ~30 min.
4. **Step 4** — User runs `docker compose -f docker-compose.demo.yml up`
   on Mac host; configures linphone; dials 9100. Listens.
5. **Step 5** — Iterate on audio quality / timing / latency until
   user is happy. Likely needs:
   - Resample 24 kHz model output → 8 kHz channel rate (Speex resampler
     on edge side, since the module also has a resampler but the model
     emits 24 kHz default — converting at the edge saves a hop).
   - Adjust `tts_jitter_buffer_ms` / `tts_preroll_ms` defaults (W6).

## Acceptance (subjective)

- [ ] User dials 9100 from linphone, hears the bot speech clearly.
- [ ] No audible glitches in the first 3 seconds (jitter buffer doing
      its job).
- [ ] FS hangup propagates to tts-edge (engine stream cancels within
      2 s of caller hanging up).
- [ ] Re-dial works (no leaked state in module from previous call).
- [ ] `docker compose down` leaves no orphaned processes.

## Out of scope

- TLS for the module ↔ edge stream (plaintext only inside compose).
- API key / mTLS auth on the module's control gRPC (demo uses
  `grpc_api_key="demo"`).
- Multi-tenant — single tenant "demo".
- STT / voicebot / recording — TTS playback only.
- Real-model voice cloning — use fixture or default `vi-VN` voice.
- Production-grade FS config (no carrier-grade SIP profile,
  no codec negotiation tuning beyond G.711μ).
