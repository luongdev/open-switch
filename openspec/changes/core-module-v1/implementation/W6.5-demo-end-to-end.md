# W6.5 — End-to-end TTS demo (caller dials → hears bot audio)

## Goal

Prove the W6 media plane end-to-end by making a SIP phone (linphone
on the user's Mac) dial into a Docker-hosted FreeSWITCH instance and
hear bot speech synthesised by the existing open-tts engine, via the
`StartBot` RPC (W7 Track D).

This is **not a wave** — it is a sample / integration smoke that
exercises the full stack. It lands after the W6.5 fix-sprint, the
W6.6 silence-driver hotfix, and the W7 Track D implementation. The
demo doc lands now (planning), the actual implementation runs in
that order.

## Architectural lens — V1 topology (per tasks.md §"V1 scope confirmed")

```text
   Linphone (Mac host)
   │   SIP REGISTER + INVITE → 127.0.0.1:5060
   │   RTP G.711μ @ 8 kHz, port 16384-32768
   ▼
   ┌────────────────────────────────────────────────────────────┐
   │ Docker compose                                              │
   │                                                             │
   │  ┌────────────────────┐    ┌──────────────────────────────┐ │
   │  │ FreeSWITCH 1.10.12 │    │  tts-edge (Go)               │ │
   │  │  + mod_open_switch │    │  port 50061                   │ │
   │  │  + mod_tone_stream │    │   open_switch.media.v1.       │ │
   │  │     (silence_stream)    │   MediaBridge.Stream adapter  │ │
   │  │                    │◀──▶│   ↓ internal Synthesize       │ │
   │  │  ext 9100:         │    │  ┌────────────────────────┐   │ │
   │  │    answer          │    │  │ tts-engine (Python)    │   │ │
   │  │    lua start_bot   │    │  │  + llama-server (or    │   │ │
   │  │    park            │    │  │     fixture mode)      │   │ │
   │  │                    │    │  └────────────────────────┘   │ │
   │  │  Module-side:      │    └──────────────────────────────┘ │
   │  │  - StartBot RPC    │                                     │
   │  │    handler attach  │                                     │
   │  │    READ_STREAM +   │                                     │
   │  │    WRITE_REPLACE   │                                     │
   │  │    bug on KH chan  │                                     │
   │  │  - SilenceDriver   │                                     │
   │  │    spawns          │                                     │
   │  │    silence_stream  │                                     │
   │  │    playback thread │                                     │
   │  │    (W6.6) to drive │                                     │
   │  │    the write side  │                                     │
   │  │                    │                                     │
   │  │  control gRPC:50071                                     │
   │  └────────────────────┘                                     │
   └─────────────────────────────────────────────────────────────┘
```

**Topology rules** (locked by W6.5 architectural decision):

- **No `mod_conference`** — single-target IVR pattern, no need for a
  conference bridge.
- **No phantom-session origination** — `StartBot` attaches bugs
  directly to the caller's channel. No fake SIP leg.
- **No `switch_ivr_eavesdrop_session`** — the multi-target bug
  attachment pattern bypasses eavesdrop entirely. The bot bugs
  carry `function_name = "mod_open_switch"`, NOT `"eavesdrop"`, so
  W7 Track A's MEDIA_BUG_START detector does not flag them.

For the demo, `StartBot` is called with **one target only** (the KH
channel). The 3-way (KH + Agent + bot) scenario is documented as an
optional extension below.

## Components to build

### 1. tts-edge `MediaBridge` adapter (open-tts repo)

**Path**: `tts-edge/internal/bridge/openswitch/`

A new bridge alongside the existing
`tts-edge/internal/bridge/freeswitch/` legacy `socket()` adapter
(which speaks raw TCP audio_socket protocol — kept for backwards
compatibility, not used by this demo). The new adapter speaks the
gRPC `open_switch.media.v1.MediaBridge` schema.

**Files** (~400 LOC):

```text
tts-edge/internal/bridge/openswitch/server.go         (gRPC service registration)
tts-edge/internal/bridge/openswitch/stream_handler.go (one bidi stream per bot)
tts-edge/internal/bridge/openswitch/synthesis_proxy.go (StartSession → StartTurn → … mapping)
tts-edge/internal/bridge/openswitch/frame_codec.go    (PCM-S16LE re-frame to 20 ms @ 8 kHz)
tests/integration/openswitch_bridge_test.go           (smoke test)
```

**Stream handler responsibilities**:

1. Receive `FromModule.start` (must be the first message; else
   error `INVALID_ARGUMENT`).
2. Branch on `start.purpose`:
   - `TTS_PLAYBACK` (W6 legacy) or `VOICEBOT_DUPLEX` (W7 Track D
     multi-target): open an internal `RealtimeTts.Synthesize`
     stream; send `StartSession{tenant_id, voice_id}`, then
     `StartTurn{}`, `AppendText{start.start_message}`, `CommitText{}`.
     `voice_id` comes from `start.variables["voice_id"]` or a
     compose-level default (e.g. `vi-VN-default`).
   - `STT_TRANSCRIBE` (W6 legacy) or `STT_LISTEN` (W7): stub —
     return `Control{error: UNIMPLEMENTED}`. The demo doesn't
     exercise STT.
   - `RECORDING_RELAY` (W7 Track B): out of scope for the demo.
3. Drain `TtsEvent.audio_frame` from the engine; re-frame to
   `start.sample_rate_hz` (typically 8000 for FS G.711);
   emit `FromService.audio = AudioFrame{seq, ts_samples,
   duration_samples, payload_pcm_s16le}` at 50 fps (20 ms each).
4. **Multi-target support**: for `VOICEBOT_DUPLEX` purpose with
   `channels > 1` or with multiple StreamStarts on one
   `bot_id` — V1 demo skips this (single target only). V1 production
   uses W7 Track D's per-target stream multiplexing where the
   server sees one stream per bot regardless of target count, and
   the module fans-out on its side.
5. Half-close on `FromModule.control{op: HALF_CLOSE_SEND}` or stream
   client cancel.
6. Map errors to gRPC status codes.

The internal `Synthesize` call reuses the existing engine path
unchanged.

### 3. Dialplan — extension 9100 (one-target IVR demo)

```xml
<extension name="osw-demo-bot">
  <condition field="destination_number" expression="^9100$">
    <action application="answer"/>
    <action application="sleep" data="500"/>
    <action application="set" data="osw_demo_message=Xin chào, đây là demo của mod_open_switch."/>
    <!-- start_bot.lua invokes StartBot RPC on the module's control gRPC.
         No <action playback="silence_stream://-1"/> needed: the W6.6
         silence driver auto-spawns when the WRITE_REPLACE bot bug
         attaches. -->
    <action application="lua" data="start_bot.lua"/>
    <action application="park"/>
  </condition>
</extension>
```

`start_bot.lua` uses `mod_lua` + `grpcurl` to invoke `StartBot` on
the module's own gRPC server:

```lua
-- start_bot.lua
local channel = session:getVariable("uuid")
local message = session:getVariable("osw_demo_message") or "Hello demo."

local payload = string.format([[{
  "header": {"tenant_id":"demo","request_id":"%s"},
  "target_channel_uuids":["%s"],
  "upstream_endpoint":"tts-edge:50061",
  "purpose":"VOICEBOT_DUPLEX",
  "sample_rate_hz":8000,
  "start_message":%q,
  "variables":{"voice_id":"vi-VN-default"}
}]], channel, channel, message)

local cmd = string.format(
  [[grpcurl -plaintext -d '%s' 127.0.0.1:50071 open_switch.control.v1.Control/StartBot]],
  payload:gsub("'", "'\\''"))

local rc = os.execute(cmd)
freeswitch.consoleLog("INFO", "start_bot.lua: StartBot rc=" .. tostring(rc) .. "\n")
-- Note: `bot_id` is returned in StartBot's response.  The Lua handler
-- doesn't capture it for the demo (we let CS_DESTROY on hangup auto-
-- stop the bot via ActiveBots::StopByChannel — W6.5 P1-003 fix).
```

(Production setups would write a tiny Go/C++ client embedded in FS
or use FS's `mod_curl_grpc`; for the demo, `grpcurl` is sufficient
and easier to follow.)

### 5. Linphone config on the Mac host

```text
SIP identity:    sip:1000@127.0.0.1:5060
Transport:       UDP
Outbound proxy:  127.0.0.1:5060
(No auth — internal profile is open in demo)
Dial:            9100
```

When the user dials 9100:

1. Linphone INVITE → docker FS port 5060 (mapped on host).
2. FS answers, runs dialplan, executes `start_bot.lua`.
3. Lua shells out to `grpcurl` calling `StartBot` with
   `target_channel_uuids=[KH_channel_uuid]`, `purpose=VOICEBOT_DUPLEX`,
   `start_message="Xin chào, …"`.
4. Module's `StartBotHandler::Handle`:
   - Validates args.
   - Attaches `SMBF_READ_STREAM` + `SMBF_WRITE_REPLACE` bugs on KH
     channel via `MediaBugManager`.
   - W6.6 SilenceDriverRegistry detects WRITE_REPLACE on a non-bridged,
     non-broadcast channel and **spawns the silence_stream
     playback thread** to drive the write side.
   - Opens bidi gRPC stream to `tts-edge:50061`. Sends `FromModule.start`
     with `purpose=VOICEBOT_DUPLEX`, `start_message="Xin chào, …"`.
5. tts-edge maps to internal `Synthesize`; engine emits audio bytes;
   tts-edge re-frames to 20 ms @ 8 kHz; sends `FromService.audio`
   frames back to the module.
6. Module's gRPC reader pushes frames through `TtsPlayoutBuffer`
   (preroll + jitter buffer); `BugFanout` routes to the one target
   queue.
7. FS write-side loop calls `OswBotWriteReplace` on the WRITE_REPLACE
   bug for every silence_stream frame. The callback pops a frame
   from the target queue and calls
   `switch_core_media_bug_set_write_replace_frame`. The replacement
   bot audio overrides the silence.
8. When the queue is empty (after the utterance ends), the callback
   returns `SWITCH_TRUE` without setting the replace frame
   (passthrough per FF-036) — the silence_stream frame flows
   unchanged. Caller hears silence until the bot speaks again.
9. Linphone receives RTP packets, decodes G.711, plays through Mac
   speakers.

User hears: **"Xin chào, đây là demo của mod_open_switch."**

When the caller hangs up:

1. SIP BYE arrives. FS state hook fires `CS_DESTROY` for the KH
   channel.
2. `on_channel_destroy` callback (W6.5 P1-003 fix) calls
   `g_active_bots_->StopByChannel(uuid)` → `BotSession::Stop()` for
   every bot on that channel → detach bugs → drain + close gRPC
   stream.
3. `MediaBugManager::DetachAll(uuid)` removes any remaining bugs.
4. `g_silence_driver_registry_->RemoveForChannel(uuid)` sets
   `CF_BREAK` on the channel (FF-037), `switch_ivr_play_file` exits,
   the silence-driver thread joins.
5. Audit events `osw.media.bot.stopped` + `osw.media.silence_driver.stopped`
   emit.

## Acceptance (subjective)

- [ ] User dials 9100 from linphone, hears the bot speech clearly.
- [ ] No audible glitches in the first 3 seconds (jitter buffer
      doing its job).
- [ ] FS hangup propagates to tts-edge (engine stream cancels within
      `bot_drain_timeout_ms` of caller hanging up).
- [ ] Re-dial works (no leaked state in module from previous call).
- [ ] `docker compose down` leaves no orphaned processes.
- [ ] No "click / silence gap" between linphone INVITE and the start
      of bot voice (W6.6 silence driver kicks in before TtsPlayoutBuffer
      preroll completes).
- [ ] Audit events `osw.media.bot.started` and `osw.media.bot.stopped`
      visible via `SubscribeEvents` (operator can verify by running
      a small subscriber).

## Optional extension — 3-way (KH + Agent + bot)

For the optional 3-way scenario:

1. Orchestrator originates the Agent leg via `Originate` RPC (W3).
2. Orchestrator calls `Bridge` to join KH + Agent legs.
3. Orchestrator calls `StartBot` with
   `target_channel_uuids=[KH_uuid, Agent_uuid]`,
   `purpose=VOICEBOT_DUPLEX`. Module attaches bugs on **both** legs.
4. Bot voice broadcasts to both. Both KH and Agent can speak (READ
   side multiplexes both into the upstream stream with `channel_uuid`
   tags).
5. `Bridge` audio still flows naturally between KH and Agent via FS's
   normal bridged-leg media path — the WRITE_REPLACE bugs passthrough
   bridge audio when the bot is silent (FF-036).

For whisper mode (bot speaks to Agent only):

`StartBot(purpose=WHISPER, target_channel_uuids=[KH, Agent],
write_target_channel_uuids=[Agent])`. KH gets a READ_STREAM bug only
(so the upstream service can hear KH's speech), but no WRITE_REPLACE
bug (so the bot's voice never reaches KH). Agent gets both bugs
(hears bot + speaks back).

## Out of scope

- TLS for the module ↔ edge stream (plaintext only inside compose).
- API key / mTLS auth on the module's control gRPC (demo uses no
  auth; production must enable).
- Multi-tenant — single tenant `"demo"`.
- STT / recording — TTS playback only for the demo.
- Real-model voice cloning — use fixture or default `vi-VN-default` voice.
- Production-grade FS config (no carrier-grade SIP profile, no codec
  negotiation tuning beyond G.711μ, no NAT helper).

## Dependencies (HARD prerequisites)

The demo MUST run AFTER the following lands on main:

1. **W6.5 fix-sprint** — 12 codex+gemini findings, especially:
   - P1-001 trampoline forwards real `switch_media_bug_t*`
   - P1-002 ActiveMediaStreams teardown ordering (and same pattern
     applies to ActiveBots when Track D lands)
   - P1-003 RegisterStateHandlers wires CS_DESTROY → DetachAll
   - P1-005 StreamClient no 5 s deadline on long-lived context
   - P2-003 Resampler wired into bug callbacks
2. **W6.6 silence driver** — the demo's clean dialplan (no manual
   `<action playback="silence_stream://-1"/>`) depends on the module
   spawning the silence driver itself.
3. **W7 Track D `StartBot`** — the demo dialplan calls `StartBot`,
   not the legacy single-target `StartTts`.

If W7 Track D is not ready, a transitional demo using the legacy
`StartTts` RPC (single-target only, no fan-out) is acceptable —
swap `start_bot.lua` for the corresponding lua that calls `StartTts`
with the same arguments minus the multi-target fields. This is the
**legacy IVR mode** preserved for backwards compatibility.
