## Summary

W6 is not ready to merge. The largest risks are media-bug lifetime ownership and bidi gRPC shutdown: several paths can either leave streams alive after hangup, free callback state while FreeSWITCH can still call into it, or block indefinitely during StopMediaStream/module shutdown.

## P1 — Blockers (ship-stopping)

- W6-P1-001 — `src/media/bug_manager.cc:176`: `OswMediaBugTrampoline` discards the `switch_media_bug_t*` and calls Track C callbacks with `nullptr` at lines 189, 210, and 217. `OswStreamingReadTap` and `OswStreamingWriteReplace` then pass that null bug into `MediaBugGetReadReplaceFrame` / `MediaBugGetWriteReplaceFrame` (`src/control/handlers/media_bug_callbacks.cc:91`, `src/control/handlers/media_bug_callbacks.cc:146`), so production callbacks either crash in FreeSWITCH or never access the media frame. Suggested fix: keep the real `bug` parameter in the trampoline and forward it to `user_cb`; add callback tests that assert non-null bug propagation.

- W6-P1-002 — `src/control/active_media_streams.cc:117`: teardown frees `write_ctx` before `bugs.clear()` detaches the WRITE_REPLACE bug. The FS media thread can still run `OswStreamingWriteReplace`, cast `user_data` to `WriteCallbackCtx`, and dereference it at `src/control/handlers/media_bug_callbacks.cc:141-156` while `write_ctx` has already been deleted. Suggested fix: detach/clear bugs before freeing `write_ctx` and `tts_buffer`; keep the callback context alive until `MediaBugManager::Detach` has fenced callback removal.

- W6-P1-003 — `src/media/bug_manager.cc:532`: `RegisterStateHandlers()` is still a stub, so the `CS_DESTROY` path required by FF-032 is never registered even though `Module::Load` calls it (`src/core/module.cc:235-237`). The file-static handler also calls `MediaBugManager::Instance().DetachAll(...)` (`src/media/bug_manager.cc:243`), not the module-owned `bug_manager_`, and it never calls `ActiveMediaStreams::RemoveForChannel` (`src/control/active_media_streams.cc:81`). Channel hangup therefore leaves ActiveMediaStreams, StreamClient threads, and bug records alive. Suggested fix: register the state hook on the real module-owned manager and route CS_DESTROY through both `active_media_streams_->RemoveForChannel(uuid)` and manager record cleanup.

- W6-P1-004 — `src/media/bug_manager.cc:477`: `DetachInternal(..., call_remove_callback=true)` removes by shared callback (`OswMediaBugTrampoline`), while every attach registers the same callback at `src/media/bug_manager.cc:351-355`. Detaching one `BugHandle` can therefore remove every module-owned FS bug on the session while the manager initially erases only one record, breaking multi-stream calls and leaving unrelated ActiveMediaStream handles stale. Suggested fix: store the exact `switch_media_bug_t*` and remove that bug with `switch_core_media_bug_remove(session, &bug)`/`MediaBugLease` semantics; reserve callback-based cleanup only for FS-driven CLOSE record erasure.

- W6-P1-005 — `src/media/stream_client.cc:61`: `Open()` sets `ClientContext::deadline` to `open_deadline_ms` and then leaves that same context on the long-lived stream; the comment at `src/media/stream_client.cc:127-137` confirms the deadline is not cancelled. All StartTts/StartStt/StartVoicebot streams opened with the default 5000 ms deadline will be cancelled around five seconds after startup. Suggested fix: do not put a short handshake deadline on the long-lived stream context; implement handshake timeout with a separate watchdog/cancel path or use a far-future call deadline plus explicit open timeout handling.

- W6-P1-006 — `src/media/stream_client.cc:235`: `Close()` joins the writer before `WritesDone()` and never calls `TryCancel()` or enforces the spec's drain timeout. A blocking `Write()` or a peer that does not finish after half-close can hang StopMediaStream/module shutdown indefinitely; additionally `ReaderLoop` owns `Finish()` (`src/media/stream_client.cc:388`) while `Close()` may concurrently call `WritesDone()` after the reader has already entered `Finish()`. Suggested fix: centralize close state so `WritesDone()` and `Finish()` each happen exactly once, add a bounded drain timeout, and call `context_->TryCancel()` before waiting past the deadline.

## P2 — Criticals (should fix before merge but not strictly blocking)

- W6-P2-001 — `src/media/bug_manager.cc:300`: `Attach()` validates duplicate purpose and stage rank under `mu_`, releases the lock for `switch_core_media_bug_add`, then registers the record later at `src/media/bug_manager.cc:369-394`. Two concurrent Start RPCs on the same channel can both pass the uniqueness/rank checks, both attach FS bugs, and then both register duplicates or violate stage order. Suggested fix: add a pending reservation under the mutex before releasing it, or re-check after FS add and immediately remove the just-added bug if the reservation no longer holds.

- W6-P2-002 — `src/media/bug_manager.cc:373`: if `BugRecord` allocation fails after `switch_core_media_bug_add` has succeeded, the code deletes `ctx` and returns without removing the FS bug (`src/media/bug_manager.cc:373-380`). That leaves FreeSWITCH holding a media bug whose `user_data` points to freed memory. Suggested fix: on every post-add failure, remove the exact FS bug before freeing `ctx`.

- W6-P2-003 — `src/control/handlers/media_bug_callbacks.cc:102`: the media callbacks never use Track B's `Resampler`; STT forwards frames at `frame->rate` directly (`src/control/handlers/media_bug_callbacks.cc:102-117`) and TTS writes server frames directly into the FS write frame (`src/control/handlers/media_bug_callbacks.cc:151-159`). The `target_rate_hz` populated by handlers (`src/control/handlers/start_stt_handler.cc:130`, `src/control/handlers/start_tts_handler.cc:174`) is never consumed, violating FF-033's one-resampler-per-direction contract and producing wrong audio whenever channel and negotiated rates differ. Suggested fix: own per-bug resampler state in the callback context and convert both read and write paths before sending/injecting frames.

- W6-P2-004 — `src/media/tts_playout_buffer.cc:94`: `Pop()` copies up to `out_cap_samples` from the front frame and then always `pop_front()`s it at line 106. If the TTS service sends batched frames larger than one FS ptime, the buffer drops the unconsumed tail, creating audible truncation and incorrect depth accounting. Suggested fix: preserve an offset into the front frame or split pushed frames into ptime-sized chunks before queueing.

- W6-P2-005 — `src/media/tts_playout_buffer.cc:169`: `SetStreamId()` and `SetTenantId()` write plain strings without `mu_`, but handlers call `Open()` before setting those labels (`src/control/handlers/start_tts_handler.cc:160-200`, `src/control/handlers/start_voicebot_handler.cc:170-238`). The StreamClient reader thread can call `Push()` immediately via `on_audio` (`src/control/handlers/start_tts_handler.cc:152-153`) and emit overrun audit while those strings are being written, which is TSAN-visible. Suggested fix: mint/set stream metadata before `Open()` or guard label reads/writes with the same mutex.

## P3 — Importants (fix in W6.5 sprint)

- W6-P3-001 — `src/control/handlers/start_tts_handler.cc:155`: all media handlers call `grpc::CreateChannel(..., InsecureChannelCredentials())` directly (`src/control/handlers/start_stt_handler.cc:112`, `src/control/handlers/start_voicebot_handler.cc:165`) instead of using an injected channel factory/channel pool from the module. This bypasses the Track C brief's factory hook, makes unit tests use real networking, and prevents later mTLS/pooling policy from applying centrally. Suggested fix: add `SetMediaChannelFactory` to the server/skeleton and pass it into handlers.

- W6-P3-002 — `src/control/handlers/health_handler.cc:60`: Health reports `snap.active_media_bugs`, but W6 never calls `Health::SetActiveMediaBugs`; `rg` shows only the test writes that field. Operators will see zero active media bugs even while streams are attached. Suggested fix: update Health from `MediaBugManager::TotalActiveBugCount()` / registry size at attach/detach or during Health RPC snapshot.

- W6-P3-003 — `src/control/handlers/start_stt_handler.cc:103`: `[SPEC-AMBIGUITY]` the W6 brief asks for Tier-2 `osw.stt.transcript` and `osw.media.tts.*` events, but this implementation emits `stt.transcript`, `media.tts.underrun`, and `media.tts.overrun` through `osw::audit::Emit` (`src/media/tts_playout_buffer.cc:198`, `src/media/tts_playout_buffer.cc:217`). `audit::Emit` prefixes every subclass with `osw.audit.` and the default classifier maps `osw.audit.*` to Tier 1 (`src/events/tier.cc:79-85`). Suggested fix: clarify whether W6 media events are audit Tier-1 or state Tier-2; if Tier-2, emit through the event plane with matching classifier rules/names.

- W6-P3-004 — `src/control/handlers/start_voicebot_handler.cc:244`: the code comments rely on `std::vector::clear()` detaching voicebot bugs in reverse order, but `ActiveMediaStreams::TearDown` simply calls `s->bugs.clear()` (`src/control/active_media_streams.cc:123`). The standard does not give this as a reliable ownership contract, and after fixing per-bug removal the intended write-before-read detach order should be explicit. Suggested fix: store named read/write handles or detach in a deliberate order.

- W6-P3-005 — `src/control/handlers/stop_media_stream_handler.cc:42`: StopMediaStream ignores `channel_uuid` and removes by `stream_id` alone. The request schema includes `channel_uuid`; without checking it, a caller with a leaked stream ID can stop a stream on another channel/tenant. Suggested fix: have `ActiveMediaStreams::Remove` validate channel UUID (and tenant if available) before teardown, while keeping unknown/mismatch idempotency explicit.

## P4 — Nits (style / docs / minor cleanup)

- W6-P4-001 — `src/control/handlers/media_bug_callbacks.cc:61`: `OswReadTapCtx` is dead code; sequence/timestamp generation uses unrelated `thread_local` counters at lines 110-114 instead. Remove the unused struct or actually use a per-stream read callback context so sequence numbers are per stream, not per FS media thread.

- W6-P4-002 — `src/media/stream_client.cc:277`: `RingPushLocked()` duplicates the inline push logic in `SendAudio()` / `SendControl()` but is never called. Remove it or route both enqueue paths through it to reduce divergence.
